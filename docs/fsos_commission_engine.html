Supabase SQL — Commission Tables

Run these in your Supabase SQL Editor. These tables connect to your existing customers, policies, and scores tables.

-- ════════════════════════════════════════ -- COMMISSION RATE TABLE -- One row per carrier/product/option/age band -- ════════════════════════════════════════ create table commission_rates ( rate_id uuid primary key default gen_random_uuid(), carrier text not null, product_name text not null, product_type text not null, -- life|fa|fia|va|vul|mf product_option text, -- Option A, Option B, etc. age_min integer default 0, age_max integer default 99, state_code text default 'ALL', gdc_rate numeric(5,4) not null, -- e.g. 0.0700 = 7.00% trail_rate numeric(5,4) default 0, -- e.g. 0.0050 = 0.50% trail_years integer default 0, is_target boolean default false, -- life target premium notes text, effective_date date not null, archived boolean default false, created_at timestamptz default now() ); -- ════════════════════════════════════════ -- COMMISSION CASES -- One row per submitted/in-progress case -- ════════════════════════════════════════ create table commission_cases ( case_id uuid primary key default gen_random_uuid(), customer_id uuid references customers(customer_id), agency_id text references agencies(agency_id), rate_id uuid references commission_rates(rate_id), carrier text not null, product_name text not null, product_type text not null, product_option text, client_age integer, state_code text default 'TX', premium numeric(12,2), -- total premium/contribution target_premium numeric(12,2), -- life target premium (if applicable) gdc_rate_used numeric(5,4), -- rate snapshot at time of case estimated_gdc numeric(12,2), -- computed: premium × gdc_rate estimated_fsa numeric(12,2), -- computed: gdc × 0.80 trail_rate_used numeric(5,4), annual_trail numeric(12,2), -- premium × trail_rate rate_missing boolean default false, -- flagged if rate not found pipeline text, -- which GHL pipeline case_status text default 'pending', -- pending|submitted|issued|paid|lapsed submitted_at timestamptz, issued_at timestamptz, actual_gdc numeric(12,2), -- actual paid by carrier (when known) actual_fsa numeric(12,2), -- actual FSA payout (when known) notes text, created_at timestamptz default now() ); -- ════════════════════════════════════════ -- HELPER FUNCTION: calculate GDC -- Call this whenever a case is created -- ════════════════════════════════════════ create or replace function calculate_case_gdc( p_product_type text, p_carrier text, p_product text, p_option text, p_age integer, p_state text, p_premium numeric, p_target_premium numeric default null ) returns table(gdc_rate numeric, trail_rate numeric, estimated_gdc numeric, estimated_fsa numeric, annual_trail numeric, rate_missing boolean) language plpgsql as $$ declare r commission_rates%rowtype; v_gdc numeric := 0; v_trail numeric := 0; v_premium numeric; begin -- find matching rate select * into r from commission_rates where product_type = p_product_type and lower(carrier) = lower(p_carrier) and lower(product_name) = lower(p_product) and (product_option = p_option or product_option is null) and p_age between age_min and age_max and (state_code = p_state or state_code = 'ALL') and archived = false and effective_date <= current_date order by effective_date desc limit 1; if r.rate_id is null then return query select null::numeric, null::numeric, null::numeric, null::numeric, null::numeric, true; return; end if; -- life: use target premium if is_target, else full premium if p_product_type = 'life' and r.is_target and p_target_premium is not null then v_premium := p_target_premium; else v_premium := p_premium; end if; v_gdc := v_premium * r.gdc_rate; v_trail := p_premium * coalesce(r.trail_rate, 0); return query select r.gdc_rate, r.trail_rate, round(v_gdc, 2), round(v_gdc * 0.80, 2), round(v_trail, 2), false; end; $$; -- ════════════════════════════════════════ -- INSERT SAMPLE RATES (from your schedule) -- Replace with your actual FFS rate sheet -- ════════════════════════════════════════ insert into commission_rates (carrier, product_name, product_type, product_option, age_min, age_max, state_code, gdc_rate, trail_rate, trail_years, is_target, effective_date) values -- Life ('Pacific Life','Pacific Horizon IUL','life','Option A',0,99,'ALL',0.9500,0,0,true,'2024-01-01'), ('Pacific Life','Pacific Horizon IUL','life','Option B',0,99,'ALL',0.9000,0,0,true,'2024-01-01'), ('Protective','Single Premium Whole Life','life',null,50,80,'ALL',0.0700,0,0,false,'2024-01-01'), -- Fixed Annuity ('Athene','Agility 10 Option 1','fa','Option 1',0,70,'ALL',0.0700,0,0,false,'2024-01-01'), ('Athene','Agility 10 Option 2','fa','Option 2',0,70,'ALL',0.0550,0.0025,10,false,'2024-01-01'), -- FIA ('MassMutual Ascend','Legend 7 Option 1','fia','Option 1',0,80,'ALL',0.0800,0,0,false,'2024-01-01'), ('MassMutual Ascend','Legend 7 Option 2','fia','Option 2',0,80,'ALL',0.0650,0.0025,10,false,'2024-01-01'), ('North American','Charter Plus 10','fia',null,0,80,'ALL',0.0750,0,0,false,'2024-01-01'), -- Mutual Fund / IRA ('Various','Mutual Fund IRA','mf',null,0,99,'ALL',0.0100,0.0050,99,false,'2024-01-01');
Make Scenario — Auto-create commission case on GHL pipeline update

When an opportunity moves to "Application Submitted" in any GHL pipeline, Make creates a commission case in Supabase automatically.

Make Scenario: GHL Opportunity Stage → Supabase Commission Case Module 1 — Trigger: GHL Webhook → Opportunity Stage Changed Filter: stage = "Application Submitted" Module 2 — Get contact data: GHL → Get Contact → use contact_id from webhook Extract: email, phone, custom fields (product_type, carrier, product_name, age, premium, option) Module 3 — Lookup customer in Supabase: HTTP GET /rest/v1/customers?email=eq.[email] → returns customer_id Module 4 — Calculate GDC: HTTP POST /rest/v1/rpc/calculate_case_gdc Body: {product_type, carrier, product, option, age, state, premium} → returns: gdc_rate, estimated_gdc, estimated_fsa, rate_missing Module 5 — Insert commission case: HTTP POST /rest/v1/commission_cases Body: {customer_id, agency_id, carrier, product, premium, gdc_rate_used, estimated_gdc, estimated_fsa, trail_rate_used, annual_trail, rate_missing, pipeline, case_status: 'pending'} Module 6 — If rate_missing = true: Send email alert: "Commission rate missing for [product] Case [case_id] excluded from forecast. Add rate to commission_rates table to include."
How the commission engine connects to the FSOS
01
Rate Table lives in Supabase — you own it
The commission_rates table holds every carrier, product, option, age band, GDC rate, and trail rate. You add rates here once. Every calculation everywhere uses the same source of truth. No hardcoded rates anywhere.
02
GHL pipeline stage triggers commission case creation
When you move an opportunity to "Application Submitted" in any of the 4 GHL pipelines, Make automatically creates a commission case in Supabase. It looks up the right GDC rate, calculates estimated GDC and FSA payout, and stores it — all without manual entry.
03
Missing rates are flagged — not estimated
If a product's GDC rate is not in the table, the case is marked rate_missing = true and excluded from all forecasts. You get an email alert. The case stays in the system but does not produce a fictional number — it waits until you add the correct rate.
04
7AM briefing includes daily revenue forecast
Your morning briefing adds: total estimated GDC in pipeline, total estimated FSA payout, cases issued this week, and trail revenue building. You see the full financial picture every morning without opening a spreadsheet.
05
Archived rates preserve historical accuracy
When FFS updates their commission schedule, you add the new rate with a new effective_date and set the old rate to archived = true. Cases submitted under the old rate keep their original calculation. New cases use the current rate. No data corruption.
06
Actual vs. estimated reconciliation
When Farmers/FFS pays out, you update actual_gdc and actual_fsa on the commission case. Over time this builds a reconciliation record showing estimated vs. actual — so you can spot discrepancies, identify products where estimates are consistently off, and know your true yield per product type.
